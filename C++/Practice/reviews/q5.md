# ❌ `areValidInt()` is still incorrect

This is the only thing I'd spend time discussing.

Your function is trying to answer:

> "Are **both** numbers integers?"

But instead it asks:

> "Is the **sum** an integer?"

Those are not the same question.

---

## Counterexample 1

```text
num1 = 5.2
num2 = 7.8
```

Their sum is:

```text
13.0
```

Your function returns:

```text
true
```

So it thinks they're valid integers.

But

```text
5.2
```

isn't an integer.

Neither is

```text
7.8
```

---

## Counterexample 2

```text
1.4
2.6
```

Sum:

```text
4.0
```

Again your function returns **true**.

Still wrong.

---

## Why this happened

I think I know your thought process.

You were thinking:

> "If I compare the real sum with the integer sum, I can detect decimals."

That works **sometimes**, but the decimal parts can cancel each other out.

For example:

```text
0.2 + 0.8 = 1.0
```

The decimals disappear.

So the sum doesn't tell you whether each input was an integer.

# Design observation

The moment I see a helper function becoming mathematically complicated just to support one operator (`%`), it makes me ask:

> "Did I choose the right data type in the first place?"

That's why I kept recommending `int` for this assignment. It lets the design stay simple.

# One more thing

Your `errorMsg()` prints:

```text
Something went Wrong!!
```

That makes debugging harder.

Instead, I'd prefer specific messages:

* `Cannot divide by zero.`
* `Modulo requires integer operands.`
* `Invalid operator.`

Specific errors help both users and future you.
